# Agente 3 — Token Unification

## Contexto

Você está trabalhando no DietForge, um dashboard nutricional React/TypeScript/Vite. Leia estes arquivos antes de começar:

- `.context/architecture.md` — estrutura completa do projeto
- `.context/css-audit.md` — auditoria de todos os valores hardcoded (gerada pelo Agente 1)

O Agente 2 já limpou código morto (5 placeholders, 11 variáveis órfãs, logs).

## O Problema

Existem **4 sistemas paralelos de variáveis CSS** no projeto, cada um redefinindo cores, backgrounds e texto com valores DIFERENTES:

| Sistema | Arquivo | Escopo | Exemplo de conflito |
|---------|---------|--------|---------------------|
| Global | `tokens.css` | `:root` | `--accent-red: #ff2a2a` |
| Macros | `dashboard-macros.css` | seletor de escopo | `--accent-red: #e63946` |
| Final | `dashboard-final.css` | seletor de escopo | `--color-accent: #e63946` |
| Supplements | `dashboard-supplements.css` | seletor de escopo | `--color-accent: #e02020` |
| Simulator | `dashboard-simulator.css` | seletor de escopo | `--accent-red: #e8293a` |

Além disso, `dashboard-presentation.css` (7.375 linhas) redefine praticamente TODAS as variáveis do `tokens.css` sob o seletor `#screen-dashboard[data-dashboard-presentation]`.

O resultado: trocar uma variável no `:root` NÃO propaga para as seções porque cada uma redefine localmente.

## Objetivo

Criar UM ÚNICO sistema de variáveis CSS autoritativo no `tokens.css`, eliminando redefinições locais desnecessárias. **O visual do site deve permanecer 100% idêntico.**

## Estratégia

A unificação segue 3 fases:

### Fase A: Inventário completo

Antes de alterar qualquer arquivo, gere uma tabela completa de TODAS as variáveis CSS declaradas no projeto, com:

```
| Variável | Valor em tokens.css | Valor em macros | Valor em final | Valor em supplements | Valor em simulator | Valor em presentation | Ação |
```

Na coluna "Ação", classifique cada variável como:

- **UNIFICAR** — todos os valores são iguais ou próximos o suficiente (diferença visual imperceptível, ex: `#e63946` vs `#e63946`). Manter apenas no `:root`.
- **PROMOVER** — variável existe apenas em seções locais, não no `:root`. Criar no `:root` com o valor mais comum.
- **MANTER OVERRIDE** — valores são intencionalmente diferentes entre seções (ex: cores de suplemento específico como `--sup-creatina-primary: #00c8ff`). Manter override local, mas documentar.
- **RENOMEAR → PADRÃO** — variável local usa nome diferente para o mesmo conceito (ex: `--color-accent` vs `--accent-red`). Renomear para o nome padronizado.

Salve esta tabela como `.context/token-unification-map.md`.

**PARE e mostre a tabela antes de executar qualquer mudança.** Eu preciso aprovar o mapa antes de você alterar arquivos.

### Fase C: Eliminar redefinições locais

Para cada arquivo de seção (`dashboard-macros.css`, `dashboard-final.css`, `dashboard-supplements.css`, `dashboard-simulator.css`, `dashboard-presentation.css`):

1. **Variáveis classificadas como UNIFICAR:** Remova a declaração local. O seletor de escopo vai herdar do `:root`.

2. **Variáveis classificadas como RENOMEAR → PADRÃO:** Substitua o nome local pelo nome padrão em TODAS as ocorrências (declaração + referências `var(--nome)`). Exemplo:
   - `dashboard-final.css` usa `--color-accent` → renomear para `--accent-primary` (em declarações E em todas as `var(--color-accent)` dentro desse arquivo)
   - Se o valor é igual ao do `:root` → remover a declaração local
   - Se o valor é diferente → manter como override local com o nome padronizado

3. **Variáveis classificadas como MANTER OVERRIDE:** Mantém a declaração local mas com o NOME PADRONIZADO. Adicione um comentário:
   ```css
   /* OVERRIDE: valor diferente do :root (#e02020 vs #e63946) — será unificado na fase Design System */
   --accent-primary: #e02020;
   ```

4. **Variáveis classificadas como PROMOVER:** Já foram adicionadas ao `:root` na Fase B. Remova a declaração local se o valor é o mesmo. Mantenha como override se diferente.

## Decisões de Nomenclatura

Use este padrão para nomes de variáveis (baseado nas sugestões do css-audit.md):

| Conceito | Nome padrão | Substitui |
|----------|-------------|-----------|
| Vermelho primário | `--accent-primary` | `--accent-red`, `--accent-lime`¹, `--color-accent` |
| Vermelho claro | `--accent-primary-bright` | `--accent-red-light`, `--color-accent-light` |
| Vermelho escuro | `--accent-primary-dark` | `--accent-red-dark`, `--color-accent-dark` |
| Vermelho dim | `--accent-primary-dim` | `--accent-red-dim` |
| Vermelho glow | `--accent-primary-glow` | `--accent-lime-glow` |
| Background base | `--bg-base` | `--bg-primary`, `--color-bg-base` |
| Background surface | `--bg-surface` | `--bg-secondary`, `--color-bg-surface` |
| Background elevated | `--bg-elevated` | `--bg-tertiary`, `--color-bg-elevated` |
| Background card | `--bg-card` | `--bg-card` (manter), `--color-bg-card` |
| Texto primário | `--text-primary` | `--color-text-primary` |
| Texto secundário | `--text-secondary` | `--color-text-secondary` |
| Texto terciário | `--text-tertiary` | `--color-text-tertiary` |
| Ouro/gold | `--accent-gold` | `--color-gold` |
| Ouro claro | `--accent-gold-light` | `--color-gold-light` |

¹ Nota: `--accent-lime` no `tokens.css` atual aponta para `#ff2a2a` (vermelho!) — é um nome errado herdado de quando o projeto usava verde-neon. Renomear para `--accent-primary`.

## Regras Críticas

1. **Visual idêntico.** Se uma seção usa `--accent-red: #e63946` e o `:root` define `--accent-primary: #ff2a2a`, NÃO remova o override local. A cor da seção deve continuar sendo `#e63946` até que o Design System decida unificar.

2. **Não toque em valores hardcoded.** Este agente só mexe em VARIÁVEIS CSS (declarações e referências `var()`). Cores hex/rgba hardcoded serão tratadas pelo Agente 4.

3. **Não toque nos seletores de escopo.** O prefixo `#screen-dashboard[data-dashboard-presentation]` e os seletores de escopo das seções devem permanecer intactos. Só as VARIÁVEIS dentro deles mudam.

4. **Não toque em variáveis de suplemento** (`--sup-creatina-*`, `--sup-vitd-*`, etc.) — são genuinamente locais e devem ser PROMOVIDAS ao `:root` mas mantidas como estão.

5. **Não remova `--accent-lime`, `--accent-violet`** e seus derivados do `:root` ainda — mesmo que os nomes sejam confusos. Eles podem ser referenciados em lugares que não vamos ver. Adicione os nomes novos como ALIAS e deprecie os antigos com comentário:
   ```css
   /* DEPRECATED — usar --accent-primary */
   --accent-lime: var(--accent-primary);
   ```
