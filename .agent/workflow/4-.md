# Agente 4 — Color Consolidation

## Contexto

Você está trabalhando no DietForge, um dashboard nutricional React/TypeScript/Vite. Leia estes arquivos antes de começar:

- `.context/architecture.md` — estrutura completa do projeto
- `.context/css-audit.md` — auditoria de todos os valores hardcoded (Agente 1)
- `.context/token-unification-map.md` — mapa de variáveis unificadas (Agente 3)
- `src/styles/tokens.css` — o tokens.css expandido pelo Agente 3 (~316 variáveis)

O Agente 3 unificou o sistema de variáveis CSS. Agora `tokens.css` tem 316 variáveis organizadas por categoria. Porém, os 12 arquivos CSS ainda estão cheios de **cores hardcoded** — valores hex, rgba, hsl escritos diretamente em propriedades CSS em vez de usar `var()`.

## O Problema

O css-audit.md identificou **~90 cores únicas**, das quais **~61 são hardcoded** (não usam variáveis). Isso significa que trocar o tema de cores do site exige encontrar e substituir manualmente dezenas de valores espalhados por ~28.000 linhas de CSS.

Exemplos do problema:
```css
/* Cor hardcoded — NÃO propaga quando muda o tema */
background: #0a0a0c;
color: #f0f0f5;
border: 1px solid rgba(255, 255, 255, 0.06);
box-shadow: 0 0 30px rgba(230, 57, 70, 0.2);

/* Cor via variável — propaga automaticamente */
background: var(--bg-base);
color: var(--text-primary);
border: 1px solid var(--border-subtle);
box-shadow: var(--shadow-accent);
```

## Objetivo

Substituir TODAS as cores hardcoded nos 12 arquivos CSS de `src/styles/` por referências a variáveis CSS existentes no `tokens.css`. **O visual do site deve permanecer 100% idêntico.**

## Estratégia

### Fase A: Mapa de substituição

Antes de alterar qualquer arquivo, gere uma tabela mapeando cada cor hardcoded encontrada nos CSS para a variável correspondente no `tokens.css`.

Para cada cor hardcoded:
1. Busque no `tokens.css` uma variável com o **valor exato** (ou equivalente normalizado — ex: `#fff` = `#ffffff` = `rgb(255,255,255)`)
2. Se existe match exato → mapeie diretamente
3. Se NÃO existe match exato mas existe uma cor **muito próxima** (diferença visual imperceptível, ex: `#f0f0f5` vs `#f0f0f0`) → NÃO substitua. Documente como "SEM MATCH — criar variável?"
4. Se a cor é usada em contexto muito específico (ex: cor de um suplemento específico) → mapeie para a variável semântica correta (ex: `--sup-creatina-primary`)

Salve como `.context/color-replacement-map.md`:

```
| Cor hardcoded | Variável tokens.css | Match | Ocorrências estimadas |
|---------------|---------------------|-------|----------------------|
| #0a0a0c | --bg-base | EXATO | ~15 |
| #0a0a0f | --bg-base | OVERRIDE (seção macros usa #0a0a0f) | ~5 |
| #f0f0f5 | --text-primary (macros) | EXATO | ~20 |
| rgba(255,255,255,0.06) | --border-subtle | EXATO | ~30 |
| rgba(200,255,0,...) | NENHUM — legado green-neon | REMOVER? | ~15 |
```

**PARE após gerar a tabela. Eu preciso aprovar antes da execução.**

### Fase B: Substituição

Após aprovação, execute as substituições em todos os 12 arquivos CSS.

**Regras de substituição:**

1. **Match exato → substituir diretamente**
   ```css
   /* ANTES */
   background: #0a0a0c;
   /* DEPOIS */
   background: var(--bg-base);
   ```

2. **Cor dentro de shorthand → substituir apenas a parte da cor**
   ```css
   /* ANTES */
   border: 1px solid rgba(255, 255, 255, 0.06);
   /* DEPOIS */
   border: 1px solid var(--border-subtle);
   ```

3. **Cor dentro de gradient → substituir cada cor individualmente**
   ```css
   /* ANTES */
   background: linear-gradient(135deg, #e63946, #ff6b76);
   /* DEPOIS */
   background: var(--protein-gradient);
   /* OU, se não existe variável de gradient: */
   background: linear-gradient(135deg, var(--protein-color), var(--protein-color-light));
   ```

4. **Cor dentro de box-shadow complexo → verificar se existe variável de shadow**
   ```css
   /* ANTES */
   box-shadow: 0 4px 24px rgba(0, 0, 0, 0.45), 0 1px 0 rgba(255, 255, 255, 0.04) inset;
   /* Se --shadow-card tem esse valor exato: */
   box-shadow: var(--shadow-card);
   /* Se NÃO tem match exato: NÃO substitua parcialmente — deixe hardcoded */
   ```

5. **Cor em rgba() com opacidade variável → NÃO substituir se a opacidade muda o propósito**
   ```css
   /* rgba(255, 42, 42, 0.12) é --accent-primary-dim? Verificar: */
   /* Se tokens.css tem --accent-primary-dim: rgba(255, 42, 42, 0.12) → SIM */
   /* Se a opacidade é diferente (0.08 vs 0.12) → NÃO substituir */
   ```

### Fase C: Legacy green-neon cleanup

O css-audit.md identificou ~15 ocorrências de `rgba(200,255,0,...)` no `screens.css`. Esta é a cor do design original (verde-neon) que foi substituída pelo tema vermelho atual.

**Analise cada ocorrência:**
- Se está dentro de um contexto que já usa variáveis vermelhas em volta (hero, form) → a cor verde provavelmente é **vestigial** e deveria ser `var(--accent-primary)` ou similar
- Se está num contexto de range/slider thumb → pode ser referenciada via `var(--range-thumb-border)` que já aponta para `var(--accent-lime)` que agora é alias de `var(--accent-primary)`
- **NÃO remova diretamente** — substitua por variável correspondente

Gere uma sublista no relatório final: "Green-neon substituições" com antes/depois/arquivo/linha.

## O que NÃO fazer

- **NÃO** crie variáveis novas no tokens.css. Use APENAS as ~316 que já existem. Se uma cor não tem match, documente e pule.
- **NÃO** altere variáveis já declaradas em tokens.css (valores, nomes, ordem).
- **NÃO** substitua cores dentro de arquivos `.tsx` ou `.ts` — apenas CSS.
- **NÃO** substitua cores dentro dos overrides locais marcados como MANTER OVERRIDE pelo Agente 3 — esses são declarações de variáveis, não usos.
- **NÃO** altere cores que fazem parte de SVG data URIs (backgrounds com `url("data:image/svg+xml,...")`).
- **NÃO** substitua `#000` / `#000000` / `black` por variável — preto absoluto é intencional em resets e sombras.
- **NÃO** substitua `#fff` / `#ffffff` / `white` por variável — branco absoluto é intencional em textos de contraste máximo e highlights.
- **NÃO** substitua `transparent`, `currentColor`, `inherit`.
- **NÃO** toque nos seletores de escopo — só os VALORES das propriedades mudam.

## Casos especiais

### Cores de macronutrientes
Proteína, carboidrato e gordura têm paletas dedicadas no tokens.css:
- `--protein-color: #e63946`, `--protein-color-light: #ff6b76`, etc.
- `--carb-color: #f4a261`, `--carb-color-light: #ffbe85`, etc.
- `--fat-color: #8b5cf6`, `--fat-color-light: #a78bfa`, etc.

Se encontrar essas cores hardcoded → substituir pela variável de macro correspondente.

### Cores de suplementos
Cada suplemento tem sua paleta (`--sup-creatina-*`, `--sup-vitd-*`, `--sup-whey-*`, `--sup-omega-*`, `--sup-mag-*`). Se encontrar essas cores hardcoded no `dashboard-supplements.css` → substituir.

### Cores TDEE
O slide TDEE tem cores específicas (`--tdee-color-bmr`, `--tdee-color-neat`, `--tdee-color-eat`, `--tdee-color-tef`, `--tdee-color-cardio`). Se encontrar essas cores hardcoded no `dashboard-presentation.css` → substituir.

### Cores de projeção/chart
O slide de projeção tem cores de chart (`--chart-weight-color`, `--chart-bf-color`, `--chart-lbm-color`, `--color-teal`, `--color-purple`). Se encontrar essas cores hardcoded → substituir.
